Pool Worker Threads v3 — Starting Notes

A short doc to align before we touch any code. We’ll expand sections as we decide things.

Goal

Move all SQLite work (hydration + advance) off the syncer’s main thread onto worker threads. Free the syncer’s event loop so it only handles WebSocket I/O, CVR PG writes, and poke delivery.

Constraints (decided)

  • 4 pool threads per syncer
  • 15-30 syncers per pod (down from 150)
  • One client group is sticky to one pool thread for life
  • Pool thread does ALL SQLite operations (hydration + advance + getRow)

Boundary

Stays on syncer main thread Moves to pool thread
WebSocket handlers Snapshotter (open SQLite, BEGIN CONCURRENT)
Auth, JWT PipelineDriver (init, advance, addQuery, removeQuery)
Client message handlers (initConnection, etc.) IVM operators (joins, filters, exists)
ViewSyncer #lock (per client group) getRow (catchup poke reads)
Replicator IPC (version-ready notifications) totalHydrationTimeMs, replicaVersion, currentVersion
CVR flush (PG writes)
Poke delivery (WebSocket sends)
TTL clock, drain coordinator

What the syncer’s main thread is doing during all of this: SQLite reads for IVM, PG reads/writes for CVR, WebSocket writes for pokes — all on the same event loop. The 'yield' markers in the generator trigger setImmediate via the global timeSliceQueue lock (view-syncer.ts:2224), which serializes these yields across ALL client groups on the syncer and gives the event loop a chance to process I/O between IVM slices.

First-time path (pipelines not yet synced)

When this.#pipelinesSynced === false (first version-ready or after a reset), the same #runInLockWithCVR callback takes a different branch (view-syncer.ts:507-525):

  1. this.#pipelines.advanceWithoutDiff() — move snapshot to head, no diff/IVM
  2. #hydrateUnchangedQueries — re-hydrate queries whose transformation hash matches cached CVR state
  3. #syncQueryPipelineSet(lc, cvr, 'missing') — hydrate the rest
  4. set this.#pipelinesSynced = true

Both addQuery and the two hydrate methods above read SQLite via the PipelineDriver/Snapshotter. They also use timer.yieldProcess() on 'yield' markers during hydration (view-syncer.ts:1373).

Key differences in what the syncer’s event loop does

Before: Inside the #lock, the syncer’s main thread synchronously iterates the generator, running IVM compute + SQLite reads inline. 'yield' markers pause the generator via setImmediate, letting the event loop process other work briefly.

After: Inside the #lock, the syncer’s main thread is awaiting the MessagePort stream. The event loop is free to process: - WebSocket I/O for OTHER client groups (their #lock is independent) - PG queries that are in flight - Timers - Messages from OTHER pool threads (for other CGs)

The key win is that the syncer’s event loop is no longer CPU-bound on IVM during an advance. It’s just buffering MessagePort reads, which is cheap.

What changed

Poke delivery On syncer (WebSocket) Unchanged — on syncer
this.#lock scope Whole #runInLockWithCVR callback: CVR load → IVM → CVR flush → poke end Same scope, but IVM step is now an await on a pool thread message

What did NOT change

  • this.#lock is still acquired before the advance and released after poke end. Client messages on the SAME client group still wait for the advance to finish.
  • CVR flush still happens AFTER IVM finishes and BEFORE pokers.end (view-syncer.ts:2111 → 2117). Client pokeEnd still waits on CVR flush.
  • Poke delivery still fans out via Promise.allSettled across connected clients; waits for the slowest.
  • The replicator notification (version-ready IPC from replicator → syncer) is unchanged.
  • The PipelineDriver class itself is unchanged — same code runs, just inside a pool thread instead of the syncer main thread. The 'yield' markers are still emitted by the generator; only the consumer changes.

Why this helps

Before: during IVM compute the syncer’s main thread is the only thing doing work. 'yield' markers briefly return control to the event loop so WebSocket I/O and PG callbacks can run, but the IVM itself still burns CPU on the main thread. With N client groups on one syncer, each advance takes its turn on the event loop and the global timeSliceQueue interleaves them in 10ms slices.

After: the syncer’s main thread is only awaiting the MessagePort stream during IVM. Other client groups’ callbacks can run freely. Pool threads run IVM at full CPU without the yield cadence. Across multiple pool threads, advances for different CGs can happen in parallel on different cores.

Claims to verify when we deploy: - Single client group wall time should not get worse (ideally similar; the IVM compute cost is the same). - Multi-client-group throughput (total advances per second per syncer) should go up when syncer was CPU-bound before. - Reset rate should go down if the elapsed times measured by the circuit breaker become more stable (because IVM no longer competes with event loop I/O on the main thread).

Communication shape

SYNCER                            POOL THREAD
  RemotePipelineDriver        ↔   PipelineDriver per client group
  (proxy implementing the            (same PipelineDriver class as stock,
   PipelineDriverInterface,           running inside a worker_thread)
   all methods async)

  Messages (syncer → pool):       Pool thread holds:
    init          (creates PipelineDriver, opens Snapshotter)
    addQuery      (calls PipelineDriver.addQuery → hydrates via SQLite)
    removeQuery
    advance       (streams results back)
    getRow        (single row from current snapshot)
    destroy       (closes Snapshotter)
                                    Map<clientGroupID, PipelineDriver>
                                    (one entry per CG assigned to this pool thread)

All methods on RemotePipelineDriver are Promise-returning. The syncer awaits each one. The stock PipelineDriver has sync methods; we need a design decision (D3) on whether to adapt the stock class or introduce a separate interface.

Streaming advance: pool thread sends advanceBegin (with version/numChanges/snapshotMs), then zero or more advanceChangeBatch messages, then advanceComplete. Syncer consumes the batches via an async iterable inside #processChanges (for await).

Why the syncer initiates everything

Triggers for all operations live on the syncer side: - addQuery / removeQuery — client WebSocket message (changeDesiredQueries) or config update - advance — {state:'version-ready'} from replicator, arriving via IPC subscription (view-syncer.ts:461) - getRow — catchup poke for a reconnecting client (WebSocket on syncer) - destroy — TTL timer or syncer shutdown (both on syncer main thread)

The CVR (source of truth for “which queries are active”) lives on the syncer in PostgreSQL. The pool thread only needs per-CG SQLite + IVM state. It doesn’t know when to advance or which queries exist until the syncer tells it.

Yield markers — reuse the same PipelineDriver

Goal: reuse the stock PipelineDriver class unchanged on the pool thread. The generator still emits 'yield' markers (via #shouldYield() → yield 'yield'). The consumer decides what to do: - Stock syncer consumer (view-syncer.ts:2001-2003): calls timer.yieldProcess() → setImmediate via timeSliceQueue. - Pool thread consumer: if (change === 'yield') continue — the pool thread has no reason to yield its event loop since it only runs IVM.

What the pool thread needs from Timer: PipelineDriver only calls timer.elapsedLap() and timer.totalElapsed() (pipeline-driver.ts:492, 665, 713, 717, 766). It never calls timer.yieldProcess(), start(), or stop() — those are consumer responsibilities. So a minimal Timer on the pool thread looks like:

function createTimer(): Timer {
  const start = performance.now();
  const lapStart = start;
  return {
    elapsedLap: () => performance.now() - lapStart,
    totalElapsed: () => performance.now() - start,
  };
}

Consequence: on the pool thread, elapsedLap keeps growing forever (no #startLap reset) and #shouldYield always returns true after the first yieldThresholdMs. That’s fine — the generator emits 'yield' markers, the pool thread consumer skips them, no harm done.

Consequence for circuit breaker: totalElapsed() on the pool thread is wall time from the start of advance. On the syncer, it was the sum of lap times (excluding yield wait). For the circuit breaker check (elapsed > totalHydrationTimeMs/2 && pos <= numChanges/2), the pool thread value is effectively pure compute time (since there’s nothing else competing on the pool thread). In stock, the syncer’s totalElapsed was also compute time (yield wait excluded). So the semantics line up.

The await problem (v2 bug)

In v2 we forgot await on getRow in the catchup poke path. must() doesn’t unwrap promises, so a Promise object got passed to contentsAndVersion which tried to destructure _0_version from it and crashed.

Prevention: - Interface declares everything as Promise<...> so TypeScript catches missed awaits - Don’t wrap pool thread results in must() — use explicit await + if (!x) throw

Mistakes from v2 to NOT repeat

  1. Missing await getRow() — fix via interface contract
  2. Per-query PipelineDriver instead of per-client-group — keep one PipelineDriver per CG
  3. Reading targetVersion on syncer, sending to pool thread — pool thread owns the version
  4. No requestId correlation — every message has requestId
