# TS → Rust Migration Status

**Source of truth.** Every change to the Rust server updates a row in this
file. "Done" means the module passes an end-to-end test against a TS-equivalent
behaviour, not that the file compiles.

Status legend:

- **Port complete** — 1:1 behavioural match with TS, verified E2E
- **Port partial** — compiles, bugs still being found against TS behaviour
- **Stub** — skeleton only; returns empty/default values
- **Missing** — nothing in Rust yet
- **Not needed** — TS-specific (multi-process, V8 isolate, etc.); replaced by a
  Rust equivalent (single-process tokio) or deliberately dropped

Every row must carry the list of *defects found so far* and a *verification
signal* (which test or manual flow proves it).

### What actually works E2E today on Rust-only

1. Initial sync of 89 tables / ~4,161 rows
2. WebSocket connect + `initConnection` + hydration pokes (~25 pokeStart cycles)
3. Sidebar renders with real data (Chat, Activity, DMs, Channels headers visible)
4. Stable connection, no poke-cookie mismatches, no infinite re-render loops

### What does not work E2E today on Rust-only

1. **Any change after startup is invisible to the client** — no change-source is
   populating `changeLog`, and the Rust change-log-consumer's schema/column
   fixes only matter once someone populates it.
2. **Mutations that go through `/zero/push`** — the wire is plumbed but no
   fan-out of results back to the client, no retry, no LMID storage.
3. **Any query with joins / correlated subqueries / IN (subquery)** — the
   pipeline driver doesn't have an operator graph, so complex queries fall back
   to `SELECT * WHERE …` with best-effort flat conditions.
4. **Schema changes on the upstream DB** — `create-table`, `add-column` etc.
   are emitted as change-tags; our consumer ignores them.

## Biggest meaningful chunks still to port (in priority order)

1. **Change source** — `services/change-source/pg/` (~6.7K TS LOC). Must read
   pgoutput from a logical replication slot, translate into the `Change` JSON
   schema, write to `changeLog`. Without this Rust-only setups cannot observe
   any change after initial sync. **This is the single biggest blocker.**

2. **ZQL pipeline driver + IVM** — `zql/src/ivm/*` + `pipeline-driver.ts`
   (~11K TS LOC). Required for joined queries to return the right rows; without
   it the app works only for queries that happen to be plain table selects.

3. **CVR diffs + catchup poke** — `services/view-syncer/cvr-store.ts` +
   `cvr.ts` (~2.4K). Needed for efficient reconnect & for `mutationsPatch` /
   `lastMutationIDChanges` to carry the right per-client state.

4. **Push/mutation fan-out** — `services/mutagen/pusher.ts` (~1.2K). Needed
   only if the app routes any mutation through Zero `push`; most Xyne flows use
   REST, so this is lower urgency than 1–3.

5. **JWT verification + write authorizer** — `auth/*.ts` (~1.2K). Must-have
   for production; currently we decode the JWT payload without verifying the
   signature.

Rough ordering by "biggest unblock per LOC spent": **1 → 2 → 3 → 5 → 4**.
