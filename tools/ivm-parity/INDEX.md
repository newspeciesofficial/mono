# ivm-parity — start here

A regression-hunting harness for Zero's IVM. **Goal: generate queries that
expose parity gaps between two IVM implementations (e.g. a reference
TypeScript one vs. an alternate Rust one) automatically, and tell you which
implementation is likely missing which behavior.**

Point the harness at two `zero-cache` instances backed by the same Postgres.
It enumerates an exhaustive bounded sample of ZQL shapes, runs each through
both instances, and reports every query whose hydration payload or
mutation-delta differs — each divergence is a real regression query you can
paste into a debugger.

## File map (ordered by how you'll hit them)

### Run and regenerate
1. `package.json` — deps (tsx, ws) and helper scripts
2. `schema.sql` + `seed.sql` + `seed-extras.sql` — PG DDL and data (apply once per fresh DB)
3. `zero-schema.ts` — Zero schema definition; the **fuzzer reads this at runtime**, so adding tables/relationships auto-extends the corpus
4. `ast-fuzz.ts` — Phase 1: bounded BFS over the AST/Condition ADT → `ast_corpus.json`
5. `seed-extractor.ts` — Phase 2: appends ~24 shipped-query archetypes to the corpus
6. `harness-coverage.ts` — hydration sweep (~5s)
7. `harness-advance-coverage.ts` — advancement/push sweep (~2 min)
8. `parity-report.ts` — combines both into `parity_report.md`

### Reference / dev-only
- `ast-render.ts` — AST → human-readable ZQL, used for the `.zql.txt` sidecar and by both reports
- `harness.ts`, `harness-advance.ts`, `harness-remove-query.ts` — legacy per-pattern harnesses (kept for single-shape debugging)
- `patterns.ts` — 43 hand-written ZQL patterns, each commented with the archetype it models
- `check-replica.ts`, `query-runner.ts`, `verify-patterns.ts`, `dump-*.ts`, `probe_*.ts` — scratch scripts from earlier debugging sessions

### Generated (gitignored)
- `ast_corpus.json`, `ast_corpus.zql.txt` — corpus from `ast-fuzz.ts` + `seed-extractor.ts`
- `coverage_run.json`, `parity_gaps.md` — hydration sweep output
- `advance_coverage_run.json`, `advance_parity_gaps.md` — advancement sweep output
- `parity_report.md` — the combined deliverable

## The regression-hunting loop

```
                              ┌────────────────────────────┐
                              │  zero-schema.ts + seed.sql │
                              │        (data + shape)      │
                              └─────────────┬──────────────┘
                                            ▼
                          ┌───────────────────────────────────┐
                          │  ast-fuzz.ts + seed-extractor.ts  │
                          │  (generate ~1000 ZQL query ASTs)  │
                          └─────────────┬─────────────────────┘
                                        ▼
                  ┌──────────────────────────────────────────┐
                  │  TS cache :4858         RS cache :4868   │
                  │       ▲                      ▲           │
                  │       │                      │           │
                  │       └──────┬───────────────┘           │
                  │              │                           │
                  │   harness-coverage.ts (5s)               │
                  │   harness-advance-coverage.ts (2 min)    │
                  └──────────────┬───────────────────────────┘
                                 ▼
                   ┌─────────────────────────────┐
                   │   parity-report.ts          │
                   │   ↓                         │
                   │   parity_report.md          │
                   │   ──────────────────────    │
                   │   - divergences by shape    │
                   │   - rendered ZQL per bug    │
                   │   - likely root cause notes │
                   └─────────────────────────────┘
```

## Hard rules (read these before trying to "fix" a divergence)

1. **Don't modify the test apparatus to hide a divergence.** The failing query is a real bug. Moving the goalposts defeats the whole point.
2. **Don't UPDATE or DELETE existing seed rows.** Add new rows via `seed-extras.sql`. Anything else breaks deterministic replay.
3. **If you fix an implementation-side bug, cite the reference-side file:line you mirrored.** No novel logic — the harness assumes one side is authoritative and the other must match its semantics.
4. **Don't edit the reference implementation** (whichever side you treat as ground truth) except with env-gated `console.log` / `tracing::debug` lines for instrumentation. Revert instrumentation after use.
5. **Re-run the full sweep after every fix** so you can confirm the diverge count actually went down and nothing else broke.

## Philosophy

Every query in `ast_corpus.json` is a potential regression test. When one
diverges, the corpus tells you *what shape* is buggy (OR-of-EXISTS,
NOT-EXISTS-on-add, flipped-join-push, …), and the INVESTIGATION-NOTES tell
you *which branch of the reference implementation* the other impl should be
mirroring. Together they let you drive divergence count to zero without
guessing.

To iterate: run, read `parity_report.md`, fix one shape, re-run. The corpus
catches regressions before they ship.
