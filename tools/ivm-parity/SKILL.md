# IVM Parity Sweep — SOP

Coverage-driven parity harness. Tests every shape of ZQL query (generated by exhaustive bounded enumeration of the AST/Condition ADT) against both the TypeScript zero-cache and the Rust zero-cache. Surfaces hydration and advancement (push) divergences in seconds.

**Hard rules — these are user-mandated and override defaults:**
- TS side gets **logs only**, gated behind `IVM_PARITY_TRACE=1`. No semantic edits to `packages/zql/`, `packages/zero-protocol/`, `packages/zero-cache/`.
- RS-side fixes must **cite exact TS file:line being mirrored**. No novel ideas. RS-only branches must be removed.
- Don't modify the test apparatus (corpus, harness, seed, patterns) to make divergences disappear. They're real bugs.
- Don't UPDATE or DELETE existing seed rows. Use `seed-extras.sql` for additive expansion only.
- Maintain streaming + behavior in both impls.

## Files (under `tools/ivm-parity/`)

| File | Purpose |
|---|---|
| `zero-schema.ts` | Authoritative schema (channels, users, participants, conversations, messages, attachments) |
| `schema.sql` | PG DDL matching `zero-schema.ts` |
| `seed.sql` | Base seed (~30 rows) |
| `seed-extras.sql` | Additive seed extension for fuller coverage (~30 more rows) |
| `ast-fuzz.ts` | Bounded BFS enumerator over the AST ADT — emits `ast_corpus.json` |
| `ast-render.ts` | AST → human-readable ZQL builder string |
| `seed-extractor.ts` | Phase 2 — appends shipped-query-shaped ASTs (source-app archetypes + zbugs-style) to the corpus |
| `harness-coverage.ts` | Hydration sweep — TS vs RS payload hash diff, ~5s for full corpus |
| `harness-advance-coverage.ts` | Push sweep — runs mutation block, diffs row deltas, ~2 min for full corpus |
| `parity-report.ts` | Combines both sweeps into `parity_report.md` |
| `harness-advance.ts` | Legacy per-pattern advance harness (43 hand-written patterns in `patterns.ts`) — kept for targeted single-bug debugging |
| `patterns.ts` | 43 hand-written ZQL patterns; legacy reference |

Generated outputs (gitignored / temporary):
| File | Source |
|---|---|
| `ast_corpus.json` | `ast-fuzz.ts` + `seed-extractor.ts` |
| `ast_corpus.zql.txt` | `ast-fuzz.ts` (human-readable rendering) |
| `coverage_run.json`, `parity_gaps.md` | `harness-coverage.ts` |
| `advance_coverage_run.json`, `advance_parity_gaps.md` | `harness-advance-coverage.ts` |
| `parity_report.md` | `parity-report.ts` (the combined deliverable) |

## Prerequisites checklist

Before running the sweep, verify:

1. **Postgres up** with `parity` database:
   ```bash
   PGPASSWORD=password psql -h 127.0.0.1 -p 6434 -U user -d parity -c 'SELECT 1' >/dev/null && echo OK
   ```
   If not, `cd apps/zbugs && npm run db-up` brings up PG container, then create the parity DB and apply schema.sql + seed.sql + seed-extras.sql.

2. **TS zero-cache running** on port 4858 with `ZERO_USE_RUST_IVM_V2=0` and `ZERO_APP_ID=parity_ts`:
   ```bash
   lsof -i :4858 | grep LISTEN || echo 'TS cache NOT running'
   ```

3. **RS zero-cache running** on port 4868 with `ZERO_USE_RUST_IVM_V2=1` and `ZERO_APP_ID=parity_rs`:
   ```bash
   lsof -i :4868 | grep LISTEN || echo 'RS cache NOT running'
   ```

4. **Replica files up to date** — both `/tmp/ivm-parity-ts.db` and `/tmp/ivm-parity-rs.db` should reflect the latest seed. If you just applied a seed change, give the replicators a few seconds to catch up.

If any prerequisite is missing, halt and ask the user — do **not** restart their caches without permission (memory: `feedback_dont_break_setup.md`).

## Running the sweep — happy path

```bash
cd /Users/harsharanga/code/mono-v2/tools/ivm-parity

# 1. Regenerate corpus (~150ms, deterministic).
#    Schema is loaded from ./zero-schema.ts dynamically — adding tables
#    or relationships there auto-extends the corpus.
npx tsx ast-fuzz.ts

# 2. (Optional) Append shipped-query seed corpus — adds ~20 hand-picked
#    archetypes inspired by real-world ZQL queries.
npx tsx seed-extractor.ts

# 3. Hydration sweep — ~5s for ~1000 ASTs, parallelism=8.
MAX_PARALLEL=8 npx tsx harness-coverage.ts

# 4. Advancement sweep — ~2 min, batch of 30 ASTs hydrate together,
#    one mutation block per batch, then per-AST delta diff.
BATCH_SIZE=30 POKE_WAIT_MS=2500 npx tsx harness-advance-coverage.ts

# 5. Combine results into one report.
npx tsx parity-report.ts

# 6. Read it.
cat parity_report.md | head -80
```

Exit codes:
- `harness-coverage.ts` and `harness-advance-coverage.ts` exit 0 only when zero divergences and zero errors. Use this for CI gating.
- `parity-report.ts` always exits 0 — it just combines reports.

## Triaging a divergence

The report groups divergences by **shape** (e.g., `EXISTS flip+scalar`, `or(or, csq)`, `NOT EXISTS`). For each shape:

1. Read the rendered ZQL from `parity_report.md` — that's the failing query.
2. Read `parity_gaps.md` or `advance_parity_gaps.md` for the per-row diff (what TS emitted vs what RS emitted).
3. Find the TS push/fetch path that produced TS's output:
   - Hydration bugs → `packages/zql/src/builder/builder.ts` and `packages/zql/src/ivm/{filter,exists,fan-in,fan-out,union-fan-in}.ts`
   - Push bugs → same files, `push` and `pushChild` methods
4. Find the RS counterpart in `packages/zero-cache-rs/crates/sync-worker/src/ivm_v2/`.
5. Apply the fix on RS side, citing `// mirrors TS <file>:<line>` in the diff.
6. Rebuild the RS bindings:
   ```bash
   cd packages/zero-cache-rs/crates/shadow-ffi && npm run build:debug
   cp *.darwin-arm64.node /Users/harsharanga/code/mono-v2/packages/zero/out/zero-cache-shadow-ffi/
   ```
7. Restart the RS cache by PID (not `pkill -f` — env vars aren't in argv):
   ```bash
   lsof -i :4868 | awk '/LISTEN/ {print $2}' | xargs kill
   # Wait for the operator to restart it, OR ask the user
   ```
8. Re-run steps 3-5 of the sweep — diverge count should drop.

## Parallelizing fixes (4-agent cap)

When you have N divergence categories to fix, dispatch up to 4 agents in parallel:
- One agent per shape category, each with the full TS+RS file paths and exact divergence sample.
- Each agent must produce a fix that cites TS file:line.
- Reject any fix that:
  - Modifies TS code (other than env-gated logs)
  - Modifies the test apparatus (`tools/ivm-parity/*`, `patterns.ts`, `seed.sql`)
  - Introduces RS code with no TS equivalent (novel idea — must be removed)
  - Uses `--no-verify` or skips hooks

## Adding new fuzz dimensions

To extend the corpus to cover more shapes:

- **More tables/relationships:** edit `zero-schema.ts` (and `schema.sql` to match). The fuzzer reads `zero-schema.ts` at runtime — new tables auto-appear.
- **More data:** append to `seed-extras.sql` (additive only). Apply with `psql … -f seed-extras.sql`.
- **More AST shape variants:** edit `ast-fuzz.ts` caps (`MAX_WHERE_DEPTH`, `MAX_BRANCHES_PER_NODE`, etc.) or add a new `cat_*` category in `whereShapes`.
- **More mutations:** edit `MUTATIONS` array in `harness-advance-coverage.ts`. Each entry is `{label, run, undo}` — `run` is the mutation SQL, `undo` is the cleanup.

Re-run the sweep after any of the above. Corpus generation is deterministic so two consecutive runs produce identical hashes (use `shasum ast_corpus.json` to confirm).
